# Presentation Structures

Use this guide when the user asks for a presentation, deck, slides, executive briefing, sponsor review, client presentation, infographic summary, or `.pptx`.

## Presentation Principle

Create a client-ready executive narrative from the PMBOK8 documentation package or live requirements. The deck should help stakeholders understand the future application, why it matters, what will be delivered, what decisions are needed, and how delivery can proceed.

Do not convert long documents into text-heavy slides. Summarize and visualize.

## Visual Style

Prefer infographic layouts:

- user journey maps
- roadmap timelines
- delivery phase timelines
- capability maps
- scope matrices
- stakeholder maps
- risk heatmaps
- assumption and decision matrices
- requirement traceability summaries
- before and after process flows
- API or integration context diagrams

Use concise headlines, short bullets, grouped visuals, clear labels, icons, and simple charts. Avoid dense paragraphs and appendix-level detail in main slides.

## Common Deck Types

## Executive Sponsor Review Deck

Use 8-12 slides.

Include:

- title and meeting context
- executive summary
- business objectives
- target users and key user journeys
- future-state solution overview
- MVP scope
- delivery roadmap
- key risks and assumptions
- decisions needed
- recommended next steps

## Client Requirements Presentation

Use 10-15 slides.

Include:

- title and purpose
- stakeholder goals
- product purpose
- user groups and journey overview
- key capabilities
- requirements themes
- sample user stories and acceptance criteria
- SBE or BDD scenario examples
- API or integration overview, if relevant
- MVP boundary
- open questions
- next-step workshop or approval path

## Future-State Application Proposal Deck

Use 10-14 slides.

Include:

- opportunity or problem statement
- target future-state vision
- proposed application overview
- user journey transformation
- core capabilities
- scope and exclusions
- technology stack assumptions or questions
- delivery approach
- indicative roadmap
- risks, dependencies, and assumptions
- decisions requested

## MVP Scope And Roadmap Deck

Use 8-12 slides.

Include:

- MVP objective
- user and stakeholder priorities
- MVP scope matrix
- out-of-scope items
- feature sequencing
- delivery roadmap
- milestone schedule
- key dependencies
- risks and mitigation
- next planning actions

## Engineering And QA Handoff Summary Deck

Use 8-12 slides.

Include:

- product and delivery context
- requirement themes
- user story and acceptance criteria coverage
- BDD or SBE scenario coverage
- API and integration summary
- data and non-functional requirement summary
- QA preparation overview
- open engineering decisions
- risks and dependencies
- next handoff actions

## Steering Committee Presentation

Use 8-10 slides.

Include:

- decision context
- current alignment summary
- scope and value summary
- roadmap and milestone view
- budget or estimate assumptions, if available
- top risks and issues
- open decisions
- change or governance needs
- requested approvals
- next steps

## Presentation Workflow

When creating a deck:

1. Identify the audience and deck purpose.
2. Select the deck type or combine compatible deck types.
3. Create a short presentation brief before generating slides.
4. Convert requirements and planning details into a storyline.
5. Choose visual patterns for each slide.
6. Generate the `.pptx` with presentation tooling when available.
7. Render or verify the deck when tooling supports it.

## Presentation Brief

Before building slides, prepare:

- audience
- objective
- primary message
- decisions requested
- suggested slide count
- source documents or live requirement areas used
- visual style notes
- assumptions and open questions

Do not expose internal discovery mechanics in the final presentation.
